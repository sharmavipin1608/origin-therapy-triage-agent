# Decision Log

Decisions made during design and implementation, in chronological order. Each entry records what was decided, why, and any alternatives that were considered.

---

## D001 — Use Anthropic SDK as the LLM provider
**Date:** 2026-06-09  
**Decision:** Use `@anthropic-ai/sdk` with a provided `ANTHROPIC_API_KEY`.  
**Reason:** The assignment provides an Anthropic API key. Model choice is not part of the rubric, so we use what is provided rather than introducing an extra dependency.  
**Alternatives considered:** OpenAI SDK — not chosen because the provided key is Anthropic.

---

## D002 — True agentic loop over hybrid rule-based routing
**Date:** 2026-06-09  
**Decision:** Claude LLM drives tool selection via a real agentic loop (item → LLM requests tools → we execute → results back to LLM → final output), rather than a rule-based dispatch table with LLM used only for parsing and drafting.  
**Reason:** More architecturally honest, handles nuanced cases (safeguarding, Spanish, clinical question, incomplete referral) without brittle if/else chains. Better demonstrates "tool orchestration" which is 25% of the rubric.  
**Alternatives considered:** Hybrid (LLM for extraction, rule-based for routing) — simpler and faster to build, but less impressive and harder to extend.

---

## D003 — Phase design docs committed to `docs/` in the repo
**Date:** 2026-06-09  
**Decision:** Each phase gets a markdown design doc under `docs/phase-N-design.md`.  
**Reason:** Keeps design decisions visible and reviewable alongside the code. The reviewer can read the design doc to understand intent, not just read the code.  
**Alternatives considered:** Design docs only in conversation — not chosen because they'd be lost after the session.

---

## D004 — `requires_human_review: true` hardcoded for all items
**Date:** 2026-06-09  
**Decision:** Set `requires_human_review = true` on every output item unconditionally.  
**Reason:** The validator enforces this for every item — it is not a judgment call, it is a hard constraint. This is a prototype triage agent; all outputs are pending human review by design.  
**Alternatives considered:** Let LLM decide per item — rejected because the validator would fail on any item where LLM returns `false`.

---

## D005 — `task_ids` collected from tool results, not from LLM output
**Date:** 2026-06-09  
**Decision:** Collect `task_ids` by capturing the `task_id` returned by each `create_task` call during the agentic loop. Do not ask Claude to output `task_ids` in its final JSON.  
**Reason:** Claude does not know the real task IDs generated by `tools.ts` (they are ULIDs created at call time). If Claude outputs them, they would be fabricated and would not match the trace.  
**Alternatives considered:** Ask Claude to output task_ids — rejected because they'd be hallucinated values with no trace backing.

---

## D006 — Enum constraints required in Anthropic tool schemas
**Date:** 2026-06-09  
**Decision:** All enum-typed parameters (`lookup_policy.topic`, `draft_message.language`, `draft_message.channel`, `create_task.assignee`, `escalate.severity`, `find_slots.discipline`) must be declared as JSON Schema `enum` arrays in the tool definitions passed to the Anthropic API.  
**Reason:** Without enum constraints, Claude will hallucinate invalid values (e.g. `"p0"` instead of `"P0"`, or `"phone_call"` instead of `"phone"`), which either crashes `dispatchTool` or produces output that fails the schema validator.  
**Alternatives considered:** Runtime coercion/normalization in dispatchTool — adds complexity and masks LLM drift; enums are the correct preventive fix.

---

## D007 — Two-layer LLM output validation
**Date:** 2026-06-09  
**Decision:** Validate Claude's final JSON response in two layers: (1) regex extraction to strip prose around the JSON block, (2) field-level validation with safe defaults for missing/invalid fields.  
**Reason:** Claude may add preamble text before `{`, output wrong enum values, or omit fields. Without validation, the agent crashes or silently writes invalid output that fails `npm run validate`. Graceful degradation with a noted fallback is better than a hard crash.  
**Alternatives considered:** Structured outputs / response format enforcement — Anthropic API does not guarantee JSON-only output without explicit prompting; prompt alone is not reliable enough.

---

## D008 — `hold_slot` only for in-network referrals with complete info
**Date:** 2026-06-09  
**Decision:** The system prompt will instruct Claude to only call `hold_slot` after a successful `find_slots`, and only for items that are in-network, have complete patient info, and are not safeguarding or clinical-question items.  
**Reason:** Calling `hold_slot` on an out-of-network or incomplete referral is an irrelevant/performative tool call, which the rubric explicitly penalizes. A slot hold for a safeguarding item is clinically inappropriate before human review.  
**Alternatives considered:** Always hold a slot for any new referral — rejected because it would be penalized and clinically wrong.

---

## D009 — Parallel processing with documented concurrency risk
**Date:** 2026-06-09  
**Decision:** Use `Promise.all` to process all 8 items in parallel for speed. Document the fallback to sequential `for...of` if the capped API key rate-limits.  
**Reason:** Parallel processing keeps end-to-end runtime under 2 minutes. The assignment explicitly notes that slow runtime is worth documenting rather than optimizing.  
**Alternatives considered:** Sequential processing — safe but slow (~2–3× longer); acceptable fallback if rate limits are hit.

---

## D010 — ESM-compatible imports throughout `agent.ts`
**Date:** 2026-06-09  
**Decision:** All local imports in `agent.ts` use `.js` extensions (e.g. `import { withItemContext } from './tools.js'`). `dotenv` loaded via `import 'dotenv/config'`.  
**Reason:** `tsconfig.json` uses `NodeNext` module resolution and `package.json` declares `"type": "module"`. Both require explicit `.js` extensions on local imports. Omitting them causes a TypeScript compile error and the agent won't run.  
**Alternatives considered:** CommonJS — the starter kit is ESM; switching would require changes to files we must not touch.

---

## D011 — Validation functions exported from `agent.ts` for unit testability
**Date:** 2026-06-09
**Decision:** `validateLlmOutput`, `validateEscalation`, `validateExtractedIntake`, `validateDiscipline`, `fallbackOutput`, `dispatchTool`, and `ValidatedOutput` are exported from `agent.ts`.
**Why:** These are the critical safety functions that prevent bad LLM output from corrupting the final JSON. They need to be unit-tested in isolation without making API calls. Exporting is the minimal change — no restructuring required.
**How to apply:** Any unit test can import these directly from `./agent.js`.

---

## D012 — Node's built-in `node:test` runner for unit tests
**Date:** 2026-06-09
**Decision:** Unit tests use `node:test` with `node --import tsx/esm --test src/agent.test.ts`.
**Why:** Zero new test dependencies. Available since Node 18, works with tsx's ESM loader.
**How to apply:** Run via `npm test`. Add new tests to `src/agent.test.ts`.

## D013 — `docs/proofs/` uses phase subfolders
**Date:** 2026-06-09
**Decision:** Proof artefacts are stored under `docs/proofs/phase-1/` and `docs/proofs/phase-2/` rather than a flat directory.
**Why:** Keeps phase snapshots distinct. Phase 1 proves structural correctness; Phase 2 proves domain correctness. A flat directory would make the two runs ambiguous.
**How to apply:** Each phase that produces a triage run gets its own subfolder with `output.json`, `tool-calls.jsonl`, and `verification-report.md`.

## D014 — Known gaps documented in verification report, not hidden
**Date:** 2026-06-09
**Decision:** The Phase 1 verification report explicitly notes that item_6 and item_8 had wrong classifications, with Phase 2 listed as the fix.
**Why:** Honesty about what Phase 1 does and does not achieve is part of the rubric. Hiding gaps by omission is worse than documenting them.
**How to apply:** Any verification report should have a "Known Gaps" section if applicable.

## D019 — Safeguarding rule must precede all other routing in the prompt
**Date:** 2026-06-09
**Decision:** The safeguarding decision block appears first in the urgency decision tree, before insurance gates and slot search logic.
**Why:** item_2's harm disclosure is buried in a routine voicemail. Prompt ordering influences model attention — placing safeguarding first ensures it is evaluated before "can we get him in for an eval" is encountered.
**How to apply:** In any future prompt revision, keep P0/safeguarding as the first block in the urgency decision tree.

---

## D020 — Insurance gate is a conditional block, not a vague prohibition
**Date:** 2026-06-09
**Decision:** The insurance gate is written as an explicit if/else: "if out_of_network → do NOT call find_slots or hold_slot."
**Why:** Vague prohibitions are unreliable. Claude interprets "be careful" as "proceed with caution." An explicit conditional naming the prohibited tools is unambiguous.
**How to apply:** Any new insurance-routing rule should name the prohibited tools explicitly, not just describe the intent.

---

## D021 — `search_patient` called only when both name AND DOB are present
**Date:** 2026-06-09
**Decision:** The system prompt requires `search_patient` only when both a child's name and a date of birth are available. Age alone (e.g. "age 5") does not qualify.
**Why:** Without this restriction, `search_patient` was called for items with no DOB (items 5, 6) — performative tool calls with no chance of a match. With it, only items 1, 3, 4, 7, 8 trigger the check.
**How to apply:** Before adding a `search_patient` call, confirm both name and exact DOB are present in the message.

## D022 — `find_slots` language parameter is mandatory for Spanish-speaking families
**Date:** 2026-06-09
**Decision:** The prompt states `find_slots` MUST include `language="es"` for Spanish-speaking families.
**Why:** Without it, Maya Chen (English-only) is returned as the first result — a provider mismatch that violates language access policy.
**How to apply:** Whenever `find_slots` is called for a Spanish-speaking family, always pass `language="es"`.

## D023 — `draft_message` for incomplete referrals goes to referring physician, not parent
**Date:** 2026-06-09
**Decision:** When parent contact is missing, the draft recipient is the referring physician.
**Why:** No parent contact exists in the referral — drafting to an unknown recipient is undeliverable. The referring physician is the only actionable contact.
**How to apply:** For any `missing_paperwork` item, the draft_message recipient is the physician name and practice from the referral body.

---

## D024 — P1 does not trigger `escalate`; only a `front_desk` task with today due date
**Date:** 2026-06-09
**Decision:** Same-day cancellation uses `create_task(assignee="front_desk", due=today)`, not `escalate`.
**Why:** `escalate` routes to `clinical_lead` and creates a formal clinical record. Same-day cancellations are operational, not clinical. Using escalate would pollute the escalation log.
**How to apply:** `escalate` is reserved for P0 only. P1 items get a front_desk task.

---

## D025 — Clinical advice prohibition includes explicit examples of prohibited content
**Date:** 2026-06-09
**Decision:** The prohibition names specific content types: milestone statements, normative comparisons, diagnostic impressions, wait-and-see recommendations.
**Why:** "Do not give clinical advice" is ambiguous. Claude may classify "R sounds typically develop by age 7" as general education, not clinical advice. Explicit examples remove the ambiguity.
**How to apply:** Any clinical-advice rule in a prompt should enumerate examples, not just state the principle.

---

## D026 — `hold_slot` pre-conditions stated as a numbered checklist
**Date:** 2026-06-09
**Decision:** The hold_slot rule is a 5-item numbered checklist — all must be true before the tool is called.
**Why:** Rules distributed across a long prompt are harder to aggregate. A checklist at the point of the instruction ensures all conditions are evaluated together, preventing partial-compliance failures.
**How to apply:** Any multi-condition gate in a system prompt should be a numbered checklist rather than scattered inline rules.

---

## D027 — Agentic loop capped at 15 iterations to prevent infinite loops
**Date:** 2026-06-09
**Decision:** The `while (stop_reason === "tool_use")` loop in `processItem` is guarded by a `MAX_ITERATIONS = 15` counter. Exceeding it throws, which feeds into the existing `catch` block and returns a `fallbackOutput` for that item.
**Why:** Without a cap, a confused model or a bad prompt revision could spin indefinitely — burning API quota and blocking the batch. The heaviest observed item (item_1) used 8 tool calls; 15 gives comfortable headroom for hidden test variants while bounding worst-case cost.
**How to apply:** If hidden test inputs have longer tool chains and hit the cap, raise `MAX_ITERATIONS` — it is a single constant at the top of the loop in `src/agent.ts`.

---

## D028 — System prompt cached with `cache_control: ephemeral`
**Date:** 2026-06-09
**Decision:** The system prompt is sent as a `TextBlockParam` array with `cache_control: { type: "ephemeral" }` rather than a plain string. This caches the prompt on the first LLM call per item; all subsequent calls in the same agentic loop reuse the cache at ~90% reduced input token cost.
**Why:** The system prompt is ~80 lines and is resent with every iteration of the loop (2–4 calls per item, 8 items). Without caching, the same tokens are billed repeatedly. The change is two lines — no architectural impact, no new failure modes.
**How to apply:** The `CACHED_SYSTEM` constant in `src/agent.ts` wraps `SYSTEM_PROMPT` from `src/prompts.ts`. If the prompt changes, the cached version updates automatically.

---

## D029 — Post-LLM safety guard enforces P0/safeguarding consistency
**Date:** 2026-06-09
**Decision:** `applySafetyGuard()` runs after `validateLlmOutput()` in the agentic loop. If any single P0 signal is present (urgency="P0", classification="safeguarding", or escalation.severity="P0"), all three are forced to agree. Overrides are noted in `decision_rationale` for auditability.
**Why:** The LLM can correctly identify a safeguarding concern but still misclassify or omit the escalation field. Relying solely on the prompt means a subtle model drift can produce a structurally valid but safety-wrong output. Code-level enforcement catches this regardless of prompt drift.
**How to apply:** Always call `applySafetyGuard(validateLlmOutput(...))` — never `validateLlmOutput` alone. The guard is a pure function and has no side effects on non-P0 items.
